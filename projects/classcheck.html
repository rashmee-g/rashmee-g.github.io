ClassCheck

A full-stack, geolocation-based attendance tracking system for university classrooms with role-based dashboards for students and professors, plus attendance reports and an exception/appeal workflow.

Context

University attendance is messy when it relies on manual roll calls, late arrivals, and unclear exceptions. ClassCheck helps instructors track attendance using real-time location verification.

Users

Students mark attendance and request exceptions. Professors manage rosters, locations, and generate reports for specific dates.

My Contributions

I led the attendance reporting + exception workflow and helped define data/schema and access control patterns with the team.

Team Setup: built collaboratively in a small team, with clearly owned use cases (ex: reports were assigned to me).

The Problem

The pain points we designed around:

Instructors need reliable records

Attendance data should be fast to collect and easy to audit. If students are late or absent, instructors need consistent rules and clean reporting.

  • Reduce “who was here?” ambiguity
  • Make reports easy to generate per day / per student
  • Keep data protected (role-based)

Students need a fair exception process

When location-based check-in fails (late bus, phone issues, room change), students need a simple appeal flow that still keeps integrity for instructors.

  • Submit an exception request with a reason
  • Instructor can approve/reject
  • Creates a clear trail for fairness

The Solution

Core flows: mark attendance → generate reports → handle exceptions.

1) Mark Attendance

Students choose the class session, the system updates geolocation, and then marks them present/absent based on classroom coordinates.

2) Create Reports

Professors request a report for a specific date; the system pulls that day’s attendance data and outputs present/absent summaries.

3) Attendance Exceptions

If needed, students submit an exception record (reason + notes). Professors review and approve/reject.

System Architecture

How the browser, APIs, auth, and database connect.

Frontend (web)

UI for students/professors with role-specific pages. Location is requested via browser geolocation, then sent to the backend for attendance marking.

HTML/CSS/JS Browser Geolocation

Backend + Services

REST APIs handle profiles, class data, attendance records, exception submissions, and report generation. Firebase enables real-time syncing for structured data.

Node.js Firebase REST

Security & Access Control

Role-Based Access + MFA Patterns.

Auth0 + MFA

Authentication uses Auth0, including multi-factor authentication for stronger account protection.

  • MFA for secure login
  • Prevents unauthorized access to attendance data

Role-Based Routing

After login, the system stores user role and routes to the correct dashboard (student vs professor). Role is set as a custom claim on the tokens so the frontend can enforce access consistently.

  • Role persisted via Auth0 post-login action
  • Namespaced claim example: https://classcheck.com/role

Database Design (Firebase Realtime DB)

Structured NoSQL schema for attendance, exceptions, classes, and user profiles.

Attendance Records

Attendance is stored by course and date, keyed by student NetID, including coordinates and check-in time.

  • Status + timestamp
  • Latitude/longitude for verification
  • Student identifiers for reporting

Exceptions (appeals)

Exception requests are stored as separate records containing the reason, notes, status, and professor/student identifiers.

  • Professor can approve/reject
  • Creates an auditable workflow

Key Flows

Geolocation Check-in

Browser collects coordinates (with permission) → backend stores location + creates a new attendance record tied to the user NetID.

Report Generation

Professor selects a date → system queries attendance for that course/date → outputs present/absent summaries that are easy to review.

Exception Review

Student submits appeal → professor dashboard lists pending exceptions → professor approves/rejects → status updates and stays traceable.

Outcomes & Learnings

What We Delivered

  • Role-based dashboards (student/professor)
  • Attendance marking with geolocation validation
  • Attendance reporting workflow (daily + student-level views)
  • Exception/appeal portal with approve/reject
  • MFA-secured login + token-based role enforcement

Next Improvements

  • Better analytics/visualizations in reports
